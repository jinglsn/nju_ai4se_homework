# SPEC_PROCESS.md — 规约与计划生成过程文档

## 一、brainstorming 关键节点

### 1.1 智能体追问的关键问题

Superpowers brainstorming 技能在本次设计中逐项追问，以下是对设计产生实质性影响的关键节点：

| 轮次 | 智能体提问 | 我的回答 | 对设计的影响 |
|------|-----------|---------|-------------|
| 1 | CLI / HTTP API / 两者都支持？ | CLI | 确定了交互模式，后续所有设计围绕 CLI 展开 |
| 2 | 面向什么类型的编码任务？ | 测试驱动修复 (C) | 将反馈闭环从"机制"变为"工作流核心"，自然融入 TDD |
| 3 | agent 能执行哪些操作？ | 标准工具集 (B) | 6 个工具覆盖完整闭环，拒绝 Git 和包管理 |
| 4 | 反馈闭环核心机制？ | 分类反馈 + 差异化策略 (B) | 8 种失败分类器成为重点深度的核心载体 |
| 5 | 治理护栏方案？ | 分级护栏 (B) | 三级状态机 + HITL，纯代码可测 |
| 6 | 记忆系统？ | 项目级记忆 (B) | 触发用户主动提供详细记忆规范 |
| 7 | 配置系统？ | 分层配置 (B) → 用户补充详细方案 | 全局+项目双层合并 |
| 8 | LLM 抽象层设计？ | 采纳接口 + 用户补充 error 和 call_history | 增强了 MockLLM 的可测试性 |
| 9 | 停机条件？ | 多条件停机 (C) | 四种停机条件，含死循环检测 |
| 10 | WebUI 方案？ | CLI + 只读仪表盘 (A) | 确定了"Web 是观察窗而非交互主体"的原则 |
| 11 | 部署平台？ | GitHub + Render + Docker | 确定了完整 CI/CD 链路 |

### 1.2 智能体提出的好问题

- **"面向什么类型的编码任务？"** — 这个问题直接决定了反馈闭环的形态。如果是通用编码任务，反馈信号分散且不确定；聚焦"测试驱动修复"后，测试结果成为唯一的客观反馈源，整个设计因此高度内聚。
- **"机制是代码还是提示词？"** — 这不是直接提问，而是贯穿整个 brainstorming 的约束。每次讨论机制时，智能体都会追问"这个能用 mock LLM 做确定性测试吗？"，这迫使我把每个设计决策落到了代码层面。

### 1.3 智能体未能提出的问题（我主动补充的）

- **记忆系统的完整规范**：智能体提出了项目级记忆的框架，但我发现其设计过于简略，主动补充了完整的记忆系统规范（存储位置、加载规则、检索约束、安全治理、审计日志、CLI 命令、测试约束、不支持的能力边界）。这是整个 SPEC 中由我主导设计的最详细模块。
- **配置系统的分层细节**：智能体提出分层方案后，我补充了核心配置域（安全治理、运行控制、记忆策略、工具权限）和硬性约束。
- **架构命名修正**：智能体呈现架构图后，我指出 `src/core/` 应改为 `src/agent_loop/`，`src/guardrails/` 应统一为 `src/guardrail/`，记忆层应从入口层下沉为全局基础组件，配置层应明确为全局基础能力。

## 二、关键迭代对话节选

### 迭代 1：技术选型（第 1 轮）

**我**：我熟悉 Python，帮我安装 Go 和 Rust，我要做反馈闭环，用 Docker，API 用自己平台的。

**智能体**：（安装 Go 和 Rust 后）确认选型为 Python + 反馈闭环 + Docker + 自有平台 API。

**决策**：Python 作为主要语言，Go 和 Rust 作为备用。反馈闭环是重点深入维度。这个阶段没有分歧，快速通过。

### 迭代 2：工具集设计（第 3 轮）

**智能体**：推荐 B（标准工具集），6 个工具——read_file、write_file、edit_file、grep、list_dir、run_shell。

**我**：追问具体有哪些工具。

**智能体**：列出 6 个工具表，解释了为什么没有 Git 和包管理——它们更适合作为护栏演示素材。

**决策**：采纳。这个判断很关键——如果工具集过大，护栏测试会变得复杂；如果过小，agent 无法完成闭环。6 个工具刚好。

### 迭代 3：记忆系统重构（第 6 轮后）

**智能体**：推荐项目级记忆 (B)。

**我**：没有直接采纳，而是主动提供了约 800 字的详细记忆系统规范，涵盖：存储内容分类、禁止存储内容、存储位置与格式、加载时机与规则、检索与上下文注入、安全与治理、实现与测试约束、不支持的能力边界。

**决策**：我的规范被完整采纳并写入 SPEC。这是我推翻智能体建议最多的环节——智能体的原始设计过于简略，我将它从"一个功能"扩展为"一个完整的子系统"。

### 迭代 4：架构命名与归属修正

**智能体**：呈现系统架构图，将记忆放在"入口层"，将主循环放在 `src/core/`。

**我**：指出四处修正：
1. `src/core/` → `src/agent_loop/`（语义更明确）
2. `src/guardrails/` → `src/guardrail/`（统一单数命名）
3. 记忆层下沉为全局基础组件
4. 配置层扩范围为全局基础能力
5. 补充停机判断与反馈闭环的联动
6. 补充工具执行的审计链路
7. 补充密钥管理模块

**决策**：全部采纳。这是"对 AI 产出进行有意义的修正"的典型案例——AI 的架构骨架正确，但命名和归属需要人工校准。

### 迭代 5：API 平台确认

**智能体**：需要确认你的 API 是 OpenAI 兼容、Anthropic 兼容还是自定义格式？

**我**：OpenAI 兼容格式，Base URL `https://njusehub.info/v1`，模型 `glm-5.2`。

**决策**：智能体据此补充了 Task 3.5（RealLLM 实现），使用 httpx 封装 OpenAI 兼容 API。

## 三、AI 建议的采纳与推翻

### 采纳的 AI 建议

| 建议 | 理由 |
|------|------|
| 测试驱动修复为核心场景 | 反馈闭环天然融入工作流，客观信号明确 |
| 6 个标准工具 | 恰好覆盖闭环，不多不少 |
| 分类反馈 + 差异化策略 | 8 种分类器是纯代码，不依赖 LLM |
| 三级护栏 | 分级状态机天然可测 |
| 多条件停机 | 死循环检测是加分项 |
| CLI + 只读 Web 仪表盘 | Web 不喧宾夺主 |
| GitHub + Render + Docker | 一条链路二用（分发+部署） |
| MockLLM 增加 error 字段和 call_history | 增强可测试性 |

### 推翻或修正的 AI 建议

| 建议 | 修正 | 理由 |
|------|------|------|
| 记忆系统简略设计 | 补充完整规范（800+ 字） | AI 的版本缺少安全治理、审计、测试约束 |
| 配置系统 | 补充核心配置域和硬性约束 | AI 只给了框架，没有具体内容 |
| `src/core/` 命名 | 改为 `src/agent_loop/` | 语义模糊，core 可以指任何东西 |
| `src/guardrails/` 命名 | 改为 `src/guardrail/` | 统一单数命名 |
| 记忆层归属 | 从入口层下沉为全局基础组件 | 被主循环主动调用，不是入口专属 |
| 缺少 RealLLM | 补充 Task 3.5 | PLAN 中遗漏了真实 LLM 实现 |

## 四、反思：brainstorming 技能的评价

### 做得好的地方

1. **逐项追问，不跳跃**：每次只问一个问题，给出 2-3 个选项，节奏控制得很好。没有任何"你确认了 A，那 B、C、D 也按默认来"的跳跃。
2. **坚持"代码不是提示词"的约束**：每次讨论机制时都会追问可测试性，这对 A 类项目至关重要。
3. **架构图呈现**：在确认所有设计决策后，用 ASCII 图呈现了完整的系统架构，让我能一目了然地发现命名和归属问题。

### 不满的地方

1. **记忆系统设计过于简略**：AI 提出的"项目级记忆"只有一句话描述，没有存储结构、检索策略、安全约束。如果我没有主动补充，这个模块会严重不足。这暴露了 AI 对"非功能性需求"的思考深度不够。
2. **未主动追问凭据安全细节**：凭据安全是作业的硬性要求，但 AI 在设计阶段没有主动追问存储方案、威胁模型、CLI 命令。这些是我在 SPEC 中自行补充的。
3. **架构命名不够严谨**：`src/core/` 和 `src/guardrails/` 的命名问题说明 AI 在命名一致性上不够敏感。
4. **缺少对"冷启动验证"的提醒**：作为 Superpowers 工作流的一部分，冷启动验证是最关键的客观证据，但 brainstorming 技能没有在设计阶段提醒我为此做准备。

### 对 Superpowers 方法论的认识

brainstorming 技能的核心价值在于**结构化追问**——它不会让你遗漏关键设计维度，但每个维度的**深度**需要你自己把握。它像一个有经验的面试官，能问对问题，但答案的质量取决于你。这与课程的命题一致：AI 守住流程纪律，但"做什么"和"做对了吗"需要人的判断力。

## 五、冷启动验证结果

### 5.1 验证方式

使用不同于 Claude Code 的 agent，全新 session，仅提供 `SPEC.md` + `PLAN.md`，不补充任何口头解释。agent 被要求从 PLAN 中选 1-2 个 task 自主实现。

### 5.2 暴露的 SPEC 缺陷

#### 缺陷 1：测试输出解析过度依赖 pytest 固定格式

`parse_test_output` 依赖正则匹配 `FAILED` 行和 `FAILURES` 分隔符，在以下场景会解析失败：
- 测试框架不是 pytest（unittest、nose）
- pytest 安装了改变输出格式的插件
- 参数化用例、带路径的用例名导致正则匹配错位
- 测试输出包含大量业务日志，干扰分隔符识别

**后果**：Agent 拿到空的失败列表，误判为"测试通过"提前停机，或完全无法定位错误点。

**严重程度**：高。反馈闭环是重点维度，解析器是闭环的入口，入口失效则整个闭环断裂。

#### 缺陷 2：死循环检测过于严格（仅匹配完全一致）

`StopJudge` 的死循环判定要求连续多轮工具调用参数完全一致，实际场景中 Agent 容易出现"换汤不换药"的无效操作：
- 反复微调 grep 关键词
- 重复读取同个文件的不同行号
- 多次 edit_file 但始终改不对核心逻辑，参数每次略有差异

**后果**：不会触发死循环判定，一直耗到 `max_iterations` 上限，属于"软卡住"。

**严重程度**：中。虽然不会无限循环，但浪费迭代轮次。

#### 缺陷 3：edit_file 工具能力不足

`edit_file` 仅支持首次精确字符串替换，实战中容易失败：
- 缩进、空格、换行符不匹配导致搜索不到目标代码
- 需要修改多处相同代码时只能改第一处
- 需要新增/删除多行代码时无法实现，只能用 write_file 全量覆写

**后果**：Agent "知道要改什么但改不了"，反复试错消耗迭代次数。

**严重程度**：中。影响 Agent 的修复成功率。

#### 缺陷 4：非交互环境下 HITL 永久挂起

`HITLStateMachine` 遇到 3 级危险操作时调用 `input()` 阻塞等待终端输入。如果 Agent 运行在 Docker 后台、CI 流水线、Web 服务等无交互终端的环境，会直接永久阻塞，没有超时降级逻辑。

**严重程度**：高。影响 Docker 分发和 CI 场景。

#### 缺陷 5：环境依赖脆弱性

- `keyring` 在精简 Docker 镜像、无桌面 Linux 环境中可能缺少密钥环后端
- 沙箱路径校验对软链接、相对路径穿越的处理有边界漏洞

**严重程度**：中。影响分发可用性。

### 5.3 Agent 主动提问/触发人工介入的场景

冷启动 agent 确认了以下需要人工介入的场景（仅安全审批和能力边界告知，无开放式追问）：

| 场景 | 行为 |
|------|------|
| 三级危险操作 | 强制确认（y/n/always） |
| 未知错误类型 | 标注"无法自动分类，由 LLM 自主分析"，连续多轮无法解决则告知用户 |
| 配置/凭据缺失 | 引导执行 `harness keyring set` 或提示 JSON 格式错误 |

### 5.4 产出质量评估

**优势**：架构分层清晰，核心机制（护栏、反馈闭环）可确定性测试，安全体系完整，工具层和 LLM 层解耦良好。

**局限**：没有 AST 静态分析能力，问题定位靠 grep 逐行搜索效率低；记忆检索是硬编码关键词匹配，覆盖面窄；代码编辑能力弱，只能做简单替换；鲁棒性不足，仅覆盖理想场景。

### 5.5 对 SPEC/PLAN 的修订

基于冷启动验证结果，对以下模块进行修订：

#### 修订 1：`parse_test_output` 增加多框架支持和容错

**修订前**：仅支持 pytest 的 `FAILED` + `FAILURES` 分隔符格式。
**修订后**：增加通用退出码判定 + unittest 格式支持 + 失败行回退解析。

#### 修订 2：`StopJudge` 死循环检测改为相似度判定

**修订前**：要求连续多轮工具调用参数完全一致。
**修订后**：改为"连续 N 轮工具调用集合相同 + 测试结果无变化"的宽松判定。

#### 修订 3：`edit_file` 增加 replace_all 和模糊匹配

**修订前**：仅首次精确匹配替换。
**修订后**：增加 `replace_all` 参数 + 行号定位编辑模式。

#### 修订 4：HITL 增加超时降级

**修订前**：`input()` 永久阻塞。
**修订后**：增加超时参数（默认 60s），超时自动拒绝；非交互环境检测，自动拒绝所有 3 级操作。

#### 修订 5：凭据存储增加加密文件降级方案

**修订前**：仅依赖 keyring。
**修订后**：keyring 失败时自动降级为 `~/.harness/credentials.enc` 加密文件存储。
